# S3 text package contract (task 5.7)

Defined **2026-05-19** on branch `feature/source-documents`. Updated **2026-05-19** for Alpha contract tightening (status/package rules, callback path validation, write ordering).

## Purpose

The **text extraction worker** reads an uploaded source document from the private documents bucket and writes a **candidate-type-free text package** to the private text bucket. Large extracted text and block-level metadata live in **S3**, not Postgres or signed callback payloads.

**Boundary index:** [source-document-data-contract-boundaries.md](./source-document-data-contract-boundaries.md) — separates text extraction, preview, and candidate extraction. Text extraction must **not** emit `candidateType` or semantic content labels; UBC/stub candidate extraction is the first step that assigns `vocab` / `expression` / `question` / `unknown`.

This document is the Alpha worker contract for:

- what the extraction worker writes to S3
- what the extraction callback sends back to Ednoda (pointers and counts only)
- what downstream **candidate extraction** receives (S3 pointers, not raw file bytes or full text in queue/webhook payloads)

**Canonical Zod schemas:** `src/schemas/source-documents/source-document-text-package.ts`  
**Extraction callback schema:** `src/schemas/source-documents/source-document-callbacks.ts` (`sourceDocumentExtractionCallbackSchema`)  
**Candidate handoff schema:** `src/schemas/source-documents/source-document-candidate-extraction.ts` (`sourceDocumentCandidateExtractionInputV1Schema`)

| Literal | Where used |
| --- | --- |
| `ednoda.extracted-source-document.v1` | `manifest.json` root `version` field |

## S3 bucket and key layout

**Bucket:** `ednoda-{env}-source-document-text` (env: `dev`, `staging`, `demo`, `prod`)

**Key prefix pattern** (all artifacts for one extraction attempt share one prefix):

```plain text
source-document-text/user/{ownerUserId}/document/{sourceDocumentId}/extraction/{extractionId}/
```

| Artifact | Key (relative to bucket root) |
| --- | --- |
| Manifest | `source-document-text/user/{ownerUserId}/document/{sourceDocumentId}/extraction/{extractionId}/manifest.json` |
| Plain text | `source-document-text/user/{ownerUserId}/document/{sourceDocumentId}/extraction/{extractionId}/plain.txt` |
| Block file | `source-document-text/user/{ownerUserId}/document/{sourceDocumentId}/extraction/{extractionId}/blocks/block-000001.json` |
| Chunk file | `source-document-text/user/{ownerUserId}/document/{sourceDocumentId}/extraction/{extractionId}/chunks/chunk-000001.json` |

**ID naming:** `{ownerUserId}` is the document owner’s user UUID. `{extractionId}` is the Postgres `SourceDocumentExtraction.id` and manifest field `document.extractionId`. Use the same numeric extraction ID everywhere.

**File numbering:** Block and chunk filenames use zero-padded six-digit sequence (`block-000001.json`, `chunk-000001.json`). Manifest `blockIndex` and `chunks` arrays must point at the written object keys.

**Ownership:** Keys are generated by Ednoda/workers from `ownerUserId`, `sourceDocumentId`, and `extractionId`. The app derives expected paths server-side (`buildSourceDocumentExtractionTextPackageKeys`); extraction callbacks validate optional worker-supplied pointers against those paths and persist the derived values.

**Server helper:** `src/utils/server/source-document/buildSourceDocumentExtractionTextPackageKeysUtil.ts`

## Package contents by terminal status (Alpha)

Alpha workers may emit only `ready`, `failed`, or `ocr_required`. The app enum still includes `partial` for Full MVP truncation; Alpha workers **must not** emit `partial`.

| Terminal status | Text package in S3 | Callback pointers |
| --- | --- | --- |
| `ready` | Full package required | Server-derived `textBucket`, `manifestKey`, `plainTextKey` (worker may echo them; must match when sent) |
| `ocr_required` | None | None — status, `extractionStrategy`, `pageCount` when known, `warnings` only |
| `failed` | None | None — `error` required |

### `ready` — full text package

```plain text
plain.txt              ← written first (with blocks/chunks)
blocks/block-*.json    ← when blockCount > 0
chunks/chunk-*.json    ← when chunkCount > 0
manifest.json          ← written last (completion marker)
```

**Write ordering (Alpha):** The worker writes `plain.txt`, block files, and chunk files first, then writes `manifest.json` last. The worker sends the extraction callback **only after** `manifest.json` has been successfully written. `manifest.json` is the package completion marker.

The manifest is the **source of truth** for block and chunk locations. Consumers load block/chunk bodies from S3 on demand.

## `manifest.json` — `ExtractedSourceDocumentManifestV1`

Root shape (partial — required sections only):

```json
{
  "version": "ednoda.extracted-source-document.v1",
  "document": { "...": "see below" },
  "extraction": { "...": "see below" },
  "outputs": { "...": "see below" }
}
```

### `document` section

Links the package to the source upload and extraction attempt:

| Field | Type | Notes |
| --- | --- | --- |
| `sourceDocumentId` | positive int | Canonical document ID |
| `extractionId` | positive int | `SourceDocumentExtraction.id` for this attempt |
| `originalFilename` | string | As uploaded |
| `originalMimeType` | string | Declared MIME |
| `fileExtension` | string | e.g. `.pdf`, `.txt` |
| `originalBucket` | string | Documents bucket name |
| `originalKey` | string | Original object key |

Example:

```json
"document": {
  "sourceDocumentId": 42,
  "extractionId": 7,
  "originalFilename": "lesson-vocab.txt",
  "originalMimeType": "text/plain",
  "fileExtension": ".txt",
  "originalBucket": "ednoda-dev-source-documents",
  "originalKey": "source-documents/user/…/original/lesson-vocab.txt"
}
```

### `extraction` section

Summary metrics and terminal extraction outcome for this package:

| Field | Type | Notes |
| --- | --- | --- |
| `status` | enum | Alpha manifest: `ready`, `failed`, `ocr_required` only. Full MVP may add `partial`. |
| `extractionStrategy` | enum | Alpha: `plain_text`, `pdf_text_layer`; Full adds `docx`, `pptx`, `csv`, `doc_converted`, `ppt_converted`, `ocr` |
| `charCount` | non-negative int | Total characters in `plain.txt` |
| `blockCount` | non-negative int | Number of entries in `outputs.blockIndex` |
| `chunkCount` | non-negative int | Number of entries in `outputs.chunks` |
| `pageCount` | positive int? | PDF page count when applicable |
| `slideCount` | positive int? | PPTX slide count when applicable |
| `tableCount` | non-negative int? | Table blocks when applicable |
| `detectedLanguages` | string[] | BCP-47 / app language codes from `ednodaLanguageCodeSchema` |
| `warnings` | string[] | Human-readable truncation or quality notes |

Example (Alpha TXT):

```json
"extraction": {
  "status": "ready",
  "extractionStrategy": "plain_text",
  "charCount": 1842,
  "blockCount": 12,
  "chunkCount": 3,
  "detectedLanguages": ["en"],
  "warnings": []
}
```

Example (Alpha PDF with insufficient text layer):

```json
"extraction": {
  "status": "ocr_required",
  "extractionStrategy": "pdf_text_layer",
  "charCount": 0,
  "blockCount": 0,
  "chunkCount": 0,
  "pageCount": 4,
  "detectedLanguages": [],
  "warnings": ["text_layer_coverage_below_threshold"]
}
```

**Manifest must not embed:** full `plain.txt` body, block `text` fields, or chunk text. Those live in separate S3 objects.

### `outputs` section

S3 pointers and indexes only:

| Field | Type | Notes |
| --- | --- | --- |
| `plainText` | `{ bucket, key }` | Pointer to `plain.txt` |
| `blockIndex` | array | One entry per block file — see below |
| `chunks` | array | One entry per chunk file — see below |

Example:

```json
"outputs": {
  "plainText": {
    "bucket": "ednoda-dev-source-document-text",
    "key": "source-document-text/user/…/extraction/7/plain.txt"
  },
  "blockIndex": [
    {
      "blockId": "block-000001",
      "bucket": "ednoda-dev-source-document-text",
      "key": "source-document-text/user/…/extraction/7/blocks/block-000001.json",
      "blockType": "paragraph",
      "charCount": 156,
      "detectedLanguage": "en",
      "importanceHint": "medium"
    }
  ],
  "chunks": [
    {
      "chunkId": "chunk-000001",
      "bucket": "ednoda-dev-source-document-text",
      "key": "source-document-text/user/…/extraction/7/chunks/chunk-000001.json",
      "index": 0,
      "charStart": 0,
      "charEnd": 512,
      "sourceBlockIds": ["block-000001", "block-000002"]
    }
  ]
}
```

**Block index item requirements** (`ExtractedSourceDocumentManifestBlockIndexItem`):

- `blockId`, `bucket`, `key`, `blockType`, `charCount` — required
- `sourcePageNumber`, `sourceSlideNumber`, `sourceRowNumber` — optional source location hints
- `detectedLanguage` — optional (`en`, `ko`, `mixed`, `unknown`)
- `importanceHint` — optional (`high`, `medium`, `low`, `unknown`)
- `blockType` — one of: `heading`, `paragraph`, `table`, `list`, `slide_text`, `speaker_notes`, `footer`, `unknown`

**Chunk pointer requirements** (`ExtractedSourceDocumentManifestChunkItem`):

- `chunkId`, `bucket`, `key`, `index`, `charStart`, `charEnd`, `sourceBlockIds` — required
- `charStart` / `charEnd` are offsets into the logical full text represented by `plain.txt`
- `sourceBlockIds` lists contributing blocks for traceability in candidate review

There is no separate Zod schema for chunk **file bodies**; chunk files are worker-owned JSON/text artifacts referenced by manifest pointers. Alpha candidate stub reads `plain.txt` and/or chunk files from S3 (master plan **10.1**).

## Block files — `ExtractedSourceDocumentBlockV1`

One JSON object per block. Version literal: `ednoda.extracted-source-document-block.v1`.

Example:

```json
{
  "version": "ednoda.extracted-source-document-block.v1",
  "sourceDocumentId": 42,
  "extractionId": 7,
  "blockId": "block-000001",
  "source": { "pageNumber": 1 },
  "blockType": "paragraph",
  "text": "What is the past tense of \"go\"?",
  "detectedLanguage": "en",
  "importanceHint": "high"
}
```

Requirements:

- `text` holds the block body (may be large — stays in S3, never in Postgres or callbacks)
- `source.pageNumber`, `source.slideNumber`, `source.rowNumber` — optional; set when the source format has pages/slides/rows
- `table` — optional `{ rows: string[][] }` for table blocks
- Manifest `blockIndex` entry must match `blockId`, `bucket`, `key`, `blockType`, and `charCount` for the same block file

---

## `plain.txt`

- UTF-8 plain text concatenation of extracted content for the attempt
- Referenced by `outputs.plainText` in the manifest
- Used by candidate extraction logic and review UI text preview
- Required when manifest `extraction.status` is `ready` or `partial`; must exist before the callback and `extraction.charCount` should reflect its length
- For `failed` or `ocr_required`, do not write a text package (no `plain.txt`, manifest, blocks, or chunks)

## Extraction callback boundary

**Route:** `POST /api/source-documents/extraction/callback`  
**Schema:** `sourceDocumentExtractionCallbackSchema`

The extraction worker **must not** send extracted text, block arrays, or chunk bodies in the callback. Ednoda persists **S3 pointers and counts** on `SourceDocumentExtraction` and updates document-level `extractionStatus`. The callback handler does **not** read S3.

### Callback rules

| Field | `ready` | `partial` | `ocr_required` | `failed` |
| --- | --- | --- | --- | --- |
| `extractionId` | Required, real `SourceDocumentExtraction.id` | Required, real `SourceDocumentExtraction.id` | Required, real `SourceDocumentExtraction.id` | Required, real `SourceDocumentExtraction.id` |
| `textBucket`, `manifestKey`, `plainTextKey` | Optional in JSON; server derives and persists canonical paths. If sent, must match `buildSourceDocumentExtractionTextPackageKeys`. | Same as `ready` | Must be omitted | Must be omitted |
| `charCount`, `blockCount`, `chunkCount` | Optional; should match manifest when provided | Optional; should match manifest when provided | Omit or zero | Omit |
| `pageCount`, `slideCount`, `tableCount` | Optional | Optional | Optional (`pageCount` when known for PDF) | Omit |
| `warnings` | Optional | Optional | Optional | Omit |
| `error` | Must be omitted | Must be omitted | Must be omitted | Required |

**Validation (app):** `completeSourceDocumentExtractionCallbackUtil` checks that `sourceDocumentId` exists, `extractionId` exists as a real prepared extraction-attempt row, the extraction row belongs to the document, terminal status transition is valid (idempotent replay of the same terminal status -> stale), and pointers match the environment text bucket and key prefix for `ready` / `partial`.

Example callback body (`ready` — pointers optional; counts only):

```json
{
  "sourceDocumentId": 42,
  "extractionId": 7,
  "status": "ready",
  "extractionStrategy": "plain_text",
  "charCount": 1842,
  "blockCount": 12,
  "chunkCount": 3
}
```

Example callback body (`ocr_required` — no package):

```json
{
  "sourceDocumentId": 42,
  "extractionId": 7,
  "status": "ocr_required",
  "extractionStrategy": "pdf_text_layer",
  "pageCount": 4,
  "warnings": ["text_layer_coverage_below_threshold"]
}
```

**Not allowed in callback:** `plainText` body, `blockIndex`, `chunks`, individual block `text`, manifest JSON blob, original file bytes, or attempt numbers masquerading as `extractionId`.

## Downstream candidate extraction boundary

After text extraction reaches `ready`, **candidate extraction** (local stub or UBC) receives **S3 pointers**, not inline text packages. This is the **first pipeline step that assigns `candidateType`** — text extraction output remains candidate-type-free. Ednoda does not hand off on `partial`, `ocr_required`, or `failed`.

**Handoff schema:** `SourceDocumentCandidateExtractionInputV1`

```json
"extractedTextPackage": {
  "manifest": { "bucket": "…", "key": "…/manifest.json" },
  "plainText": { "bucket": "…", "key": "…/plain.txt" },
  "chunksPrefix": { "bucket": "…", "prefix": "…/chunks/" },
  "blocksPrefix": { "bucket": "…", "prefix": "…/blocks/" }
}
```

Rules:

- Workers/teams **fetch** manifest, plain text, blocks, and chunks from S3 using IAM credentials
- Queue messages, Step Functions task input, and webhook payloads stay small: IDs, target context, callback URL, optional `taskToken`, and pointer objects only
- Candidate **output** (`SourceDocumentCandidateExtractionOutputV1`) returns normalized candidate rows via the node-candidates callback — not a rewrite of the text package
- Alpha stub reads `plain.txt` / chunks from S3 directly (master plan **10.1**)

---

## Postgres boundary — no block table

**There is no `SourceDocumentExtractedBlock` table.** Block-level extraction metadata is **not** stored in Postgres.

Postgres stores on `SourceDocumentExtraction`:

- `textBucket`, `manifestKey`, `plainTextKey`
- aggregate counts (`charCount`, `blockCount`, `chunkCount`, …)
- status, strategy, warnings, errors

Block bodies and per-block metadata live in S3 (`manifest.json` + `blocks/*.json`). Review UI loads them on demand via authenticated app routes or presigned GET URLs.

**Do not create `SourceDocumentExtractedBlock`.**

## Alpha scope `[Alpha]`

| Input types | Strategies | Notes |
| --- | --- | --- |
| `.txt` | `plain_text` | Paragraph blocks and optional chunks |
| `.pdf` (embedded text layer) | `pdf_text_layer` | No OCR in Alpha; insufficient coverage → `ocr_required` |

Alpha extraction worker responsibilities:

1. Read original from documents bucket after malware gate passes
2. On `ready`: write full text package under the key pattern above (`manifest.json` last), then POST signed extraction callback with counts (pointers optional; must match server paths if sent)
3. On `ocr_required` or `failed`: POST signed callback with terminal status and metadata only — no S3 text package

Out of Alpha scope (documented for later):

- DOCX, PPTX, CSV extraction
- OCR / Textract for scanned PDFs
- Worker limit truncation (`MAX_EXTRACTION_CHARS`, `MAX_BLOCKS`, `MAX_CHUNKS`, `MAX_PROCESSING_SECONDS`) — see [worker-limit-env-vars.md](./worker-limit-env-vars.md)
- `partial` status from worker limits (Full MVP **9.6**–**9.7**); Alpha workers must not emit `partial`

## Out of scope (this feature area)

- Root textbook uploads (`textbookUnitId = null`) — rejected everywhere
- Storing extracted text or block arrays in Postgres
- Embedding large payloads in Step Functions execution input (use `SourceDocumentWorkflowInputV1` — original S3 pointer + metadata only)
- Public unauthenticated S3 GET; serve artifacts via signed URLs or app routes
